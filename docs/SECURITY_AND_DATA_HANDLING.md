# セキュリティ・データ取扱いガイド（v1.0.0）

OCR Crafterのセキュリティ特性とデータの取扱いを、利用者・管理者向けにまとめたものです。実装仕様の詳細は [22_SECURITY_AND_AUDIT.md](22_SECURITY_AND_AUDIT.md) を参照してください。

## 1. ローカル処理と外部通信

- OCR Crafterは**ローカル環境で完結**します。OCR画像・ラベル・モデル・評価結果・レポートが外部Webサービスへ送信されることはありません
- **PDF生成もローカル**です（matplotlibによる生成。外部変換サービス不使用。日本語フォントはOSのフォントを使用）
- 外部への通信が発生し得るのは、**利用者が明示的に行う導入・更新作業のみ**です（`git clone/pull`・`pip install`・`npm install`、およびPaddleOCR公式モデルを未取得の状態で使う場合のモデルダウンロード）。日常のOCR業務では外部通信は発生しません
- CORS許可はローカルのVite開発サーバー（既定 `http://localhost:5173`）のみです

## 2. データの保存場所と性質

| データ | 場所 | 備考 |
|---|---|---|
| OCR画像（元画像・前処理済み） | `data/projects/<id>/raw/` `interim/` `processed/` | 元画像は前処理で変更されない（回転操作のみ例外） |
| 学習データ / 評価データ | `data/projects/<id>/outputs/ocr_dataset/` 等 | 学習画像は必ず元画像から切り出される |
| モデル | `data/projects/<id>/models/` | 管理No（M0001）で追跡 |
| ログ（Job・学習） | `data/jobs/` `data/projects/<id>/logs/` | スタックトレースはサーバー側ログのみ（画面へ出さない） |
| 監査ログ | `data/audit/audit.jsonl` | **追記型・削除/編集APIなし** |
| バックアップ | `data/backups/` | Manifest v2（SHA-256つき） |
| レポート | `data/reports/<project_id>/` | ファイルごとのSHA-256を記録 |

- **個人情報・機密情報**: 取り込む画像・ラベル・レポートに含まれる情報はすべてローカルディスクに保存されます。機密度の高い画像を扱うプロジェクトでは、保存先ディスクのアクセス権限（OSレベル）と、バックアップZIPの保管場所を管理してください
- 監査ログには**パスワード・トークン・APIキー等の機密キーを保存しない**サニタイズが強制されます（該当キーは値ごと除去・画像バイナリは保存されない・巨大値は切り詰め）

## 3. アクセス制御（権限）

- ユーザー識別は `X-Operator`（操作者名）/ `X-Role`（viewer / operator / approver / admin）ヘッダで行います。**SSO・パスワード認証は未実装**です
- 既定は「認証未設定モード（Admin互換）」で、画面にバナーで明示されます
- 本番では `OCRC_ALLOW_UNAUTHENTICATED_ADMIN=false` を設定してください: X-Operatorなしの変更系操作は401、不正Roleは403、Role未指定はviewer（最小権限）になります
- ロール別の許可操作は [ADMIN_GUIDE.md](ADMIN_GUIDE.md#2-ユーザーと権限) の表を参照
- 社内ネットワーク公開時はリバースプロキシでヘッダを付与し、クライアントからのヘッダ偽装を遮断する構成を推奨します（構成例: [24_DEPLOYMENT_GUIDE.md](24_DEPLOYMENT_GUIDE.md)）

## 4. 監査

- 24操作（プロジェクト作成/削除・学習・評価・リリース昇格/Override/Rollback・バックアップ・レポート生成/削除等）が監査ログへ記録されます
- 記録項目: Audit ID（AUD-000001）・日時・操作者・ロール・操作・対象・Before/After差分・理由・Client情報（IP・User-Agent）
- **削除・編集する機能は存在しません**（追記型）。整理はデータ保持設定（保持日数）でのみ行えます

## 5. ファイルパス・入力の安全対策（実装済み）

| 対策 | 内容 |
|---|---|
| ディレクトリトラバーサル防止 | レポートダウンロードは `data/reports` 配下限定（resolve検証）。モデル削除は `models/` 配下限定・出力削除は `outputs/` 配下限定の安全ガード |
| 削除の安全ガード | `safe_rmtree`（空パス・カレントディレクトリ・プロジェクトルートの削除を拒否） |
| 安全なファイル名 | レポート等の生成ファイル名は禁止文字を除去し、重複時は連番付与 |
| Atomic Write | レジストリJSON（jobs/releases/experiments/reports等）は一時ファイル→置換で書き込み、途中クラッシュでも破損ファイルを残さない |
| HTMLエスケープ | フロントエンドはReactの標準エスケープで表示（ラベル・ログ等のユーザー入力をHTMLとして解釈しない） |
| エラー情報の最小化 | APIエラーは統一形式（error_code/message）。スタックトレース・内部パスは画面へ返さない |
| ハッシュ | バックアップ（全ファイル）・レポート（生成物）にSHA-256を記録し、改ざん・破損を検知可能 |

## 6. 削除時の注意

| 操作 | 影響 | 保護 |
|---|---|---|
| プロジェクト削除 | 画像・ラベル・モデル・記録がすべて削除（取り消し不可） | admin権限＋プロジェクト名の入力一致 |
| モデル削除 | モデル実体とメタが削除（管理Noは再利用されない） | admin権限＋「DELETE」入力・models配下限定ガード |
| レポート削除 | メタデータと出力ファイルが削除 | 確認ダイアログ＋監査記録（report_delete） |
| データ保持設定の適用 | 期限切れの終端Job・古い監査ログを削除 | admin権限＋確認ダイアログ＋削除の事実を監査記録 |
| バックアップからの復元 | **既存プロジェクトを上書きしない**（常に新Project IDへ） | 復元前後のSHA-256検証 |

削除前のバックアップ取得を推奨します（[BACKUP_AND_RESTORE.md](BACKUP_AND_RESTORE.md)）。
